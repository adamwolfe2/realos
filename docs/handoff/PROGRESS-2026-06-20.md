## Phase 1 — Close the 2 P0 launch blockers (Tier-1)

### P0-1 · Marketplace lead PII leak  — branch `fix/p0-marketplace-pii`
- **Bug:** `/marketplace/<id>` rendered each lead's real gender + income range to
  unauthenticated visitors / non-buyers. `getFullLead` was called unconditionally
  (`page.tsx:22`), and the Income/Gender/Age rows returned the real value in BOTH
  the `owned` and `!owned` ternary branches (`page.tsx:261-278`).
- **Fix (architectural, not just cosmetic):**
  - New `getMaskedLead(id)` in `lib/marketplace/repo.ts` — returns browse-safe fields
    + existence booleans (`has.gender/income/company/...`) but NEVER the raw values.
  - `page.tsx` now computes purchase ownership FIRST (`getBuyerPurchaseForLead`), fetches
    masked always, and fetches full-PII `getFullLead` ONLY when `owned`. Raw gender/income/
    company/contact can no longer reach the render layer for a non-owner.
  - Income/Gender rows now mask to "••• (revealed on purchase)" when `!owned`; teaser rows
    driven by `has.*` existence flags. Removed now-unused `prisma` import + redundant status query.
- **Test:** `__tests__/marketplace-pii-masking.test.ts` — asserts masked view contains no raw
  PII (serialized-string scan of 9 secrets), existence flags correct, full path still returns PII.
- **Product note for Adam (non-blocking):** `age` is left visible. It already ships publicly via
  `BrowseLead` (browse cards + masked detail header) by existing product design, so masking it
  only on the detail rows would be inconsistent security-theater. Truly hiding age = remove it
  from the public browse teaser too — a product call (is age part of the public teaser?).
- **Verification:** tsc 0 · lint 0/0 · build 0/0 · **974/974** tests · security-reviewer CLEAN
  (traced every JSX PII ref) · read-only prod sanity: **2000 leads, 2000 w/income, 1985 w/gender,
  1998 AVAILABLE** — leak was live.
- **Review:** Codex hung (~8 min, exit 144 — known ChatGPT-account model stall); shipped on the
  CLEAN Claude security review + green gate (no CRITICAL/HIGH surfaced = not a blocker per contract).
- **Shipped:** PR #143 → squash-merged `70da163e` → **prod deploy SUCCESS** (live on www.leasestack.co).
  `log-mistake` appended (sibling-branch-parity class).

### P0-2 · Cal.com webhook cross-tenant injection — branch `fix/p0-cal-webhook-token`
- **Bug:** webhook `app/api/webhooks/cal/[orgId]/route.ts` treated the URL `orgId` (an org cuid) as a
  secret, but that cuid was leaked by the unauthenticated CORS-`*` chatbot config endpoint
  (`config/route.ts:239` `orgId: org.id`). Attacker: read tenant slug → get orgId → forge
  `BOOKING_CREATED` into ANY tenant's pipeline (fake leads/tours + operator emails on our Resend domain).
- **Decisive finding (read-only prod):** the Cal webhook has **ZERO live usage** (0 cal tours, 0 cal
  leads, no org) → safe to do the clean end-state (token route + delete orgId route) with no live
  tenant (SG included) broken and no external Cal reconfiguration needed.
- **Fix (token-as-path-secret, mirrors the blessed `cursive/[token]` pattern):**
  - Schema: `Organization.calWebhookToken String? @unique` + additive migration
    `20260620_add_cal_webhook_token` (nullable, `IF NOT EXISTS`, unique index — zero-downtime;
    `vercel-build` runs `prisma migrate deploy`).
  - `lib/integrations/cal-webhook.ts`: `getOrCreateCalWebhookToken` (lazy, **race-safe** via
    `updateMany` guarded on `calWebhookToken: null`), `buildCalWebhookUrl`, `CAL_TOKEN_RE` (32-hex),
    and `processCalBooking` (extracted lead/tour upsert — every write scoped to the token-resolved org).
  - New `app/api/webhooks/cal/[token]/route.ts`; **deleted** the `[orgId]` route.
  - Removed `orgId: org.id` from `config/route.ts` (no client consumer — confirmed by grep).
  - `settings/integrations` page mints the token + renders the token URL (authenticated surface only).
- **Test:** `__tests__/cal-webhook-token.test.ts` (9) — token-format guard rejects cuids/malformed,
  mint idempotency + race guard, `processCalBooking` scoping to resolved org, trigger handling.
- **Verification:** tsc 0 · lint 0/0 · build 0/0 · **983/983** tests · security-reviewer **APPROVE / LOW**
  (no CRITICAL/HIGH/MEDIUM; confirmed no remaining public token/org.id exposure, zero-downtime
  migration, race-safe mint). HMAC deferred (per-webhook secret is operator-side) — noted, not a blocker.
- **Review:** Codex hung again (exit 144, ChatGPT-account stall); shipped on security-reviewer
  APPROVE + green gate (no CRITICAL/HIGH surfaced).
- **Shipped:** PR #144 → squash-merged `7097b7f8` → **prod deploy SUCCESS** (migration applied via
  `prisma migrate deploy`). Post-deploy read-only check: `calWebhookToken` column (nullable) + unique
  index confirmed present on prod. `log-mistake` appended (path-as-secret + identifier-leak class).

- **Bug:** operator/AI content injected raw via `JSON.stringify` into `<script type="application/ld+json">`
  on public tenant pages → `</script>` breakout → arbitrary JS in every visitor's browser.
- **Fix:** new `lib/seo/serialize-json-ld.ts` — `serializeJsonLd` (escapes `<>&` + U+2028/9) for direct
